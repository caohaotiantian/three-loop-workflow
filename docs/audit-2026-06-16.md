# Multi-lens audit — three-loop-workflow skill v1.5.0

**Date:** 2026-06-16
**Branch audited:** `skill-v1.5-compliance-hardening` (HEAD `fa123be`)
**Method:** 7 independent audit lenses run in parallel, **every finding re-checked by a fresh
adversarial verifier** (instructed to refute), then a completeness critic + an 8-way gap-fill pass.
55 subagents, ~1.46M tokens. Each lens ran the repo's own gates (`check-consistency.sh`,
`check-workflow-syntax.sh`, `validate-commit-msg.sh`) and the zip↔source diff to ground findings in
real behaviour.

Severities below are the **verifier-corrected** values (2 findings were refuted outright, 18
downgraded). Per-lens findings are verified; **gap-fill** findings are critic-directed single-pass
leads (high-confidence but not independently re-verified — marked `[gap-fill]`).

## Headline

The skill is internally healthy: the shipped `SKILL.md`, the L3 engine control flow, and the live
gates all work; the committed `.skill` zip and the installed copy are byte-for-byte in sync **today**.
The material problems cluster in one place — **the skill's own enforcement machinery claims
protections it does not actually provide.** For a skill whose thesis is "mechanically protect the
discipline," that is the most important finding class, and three top items are exactly that (H1, H3,
M5).

Corrected-severity histogram (per-lens confirmed+partial): high 1 · medium 6 · low 12 · nit 18.
Gap-fill adds: high 3 · medium 6 · low 6 · nit 1.

## High severity

### H1 — Consistency gate's "two-generation termination" check is a comment-only no-op `[gap-fill]`
- **Files:** `three-loop-workflow/references/check-consistency.sh:36-38`; `CLAUDE.md:51-52`;
  `docs/design/2026-06-15-skill-v1-5-compliance-hardening.md` AC-G1 (323-325), AC-G3 (330-333).
- Lines 36-38 are a bare comment `# Two-generation termination wording.` over two `require` calls
  that grep `zero severe` / `zero general` **in `SKILL.md` only**. Those are the *per-round
  cleanliness predicate* (mermaid edges, pass-conditions), **not** the two-generation termination
  rule. The canonical `two-generation` token appears in 10+ files (SKILL.md, schemas.md, loop-1/2,
  escalation-rules, loop-3-*, multi-voter-review, l3-phase.js) with **zero** mechanical pairing. The
  gate exits 0 even if `two-generation` is dropped from every file — i.e. the discipline's core
  termination guarantee has no protection from exactly the regression class this gate exists to catch.
- Three-way overclaim: `CLAUDE.md:51-52` lists "the two-generation termination wording" among the
  gate's paired tokens (false); AC-G1/AC-G3 assert the gate enforces it "across their paired sites"
  (false), and because AC-G3 defers to AC-G1, the regression-guard intent silently evaporates.
- **Fix:** add a real paired `require "two-generation" <files...>` line and rename the comment to
  distinguish the cleanliness predicate from the termination rule — **or** correct all three docs to
  stop claiming coverage. Note `SKILL.md:28` uniquely phrases it "two consecutive clean rounds", so a
  single-token grep won't cover that site; decide token vs. exemption.

### H2 — `validate-commit-msg.sh` never validates the subject when a body is present
- **File:** `three-loop-workflow/references/validate-commit-msg.sh:34` (confirmed, high).
- Greedy `.*-m` matches through to the **last** `-m`. `git commit -m "bad subject" -m "chore: ok"`
  → the body is extracted, the real subject is never linted → **exit 0 ACCEPT**. Reproduced live.
  This is the multi-`-m` shape used throughout this repo; the bug also bypasses the phase-scope
  grammar rule (the hook's strongest purpose) whenever a body is present. Verifier: "arguably
  under-stated."
- Mitigating: the hook self-describes as an optional lint backstopped by the L3 review corner — keeps
  it high, not critical.
- **Fix:** anchor extraction to the **first** `-m` / a flag boundary (awk split on `-m`, take first
  token), then validate that token.

### H3 — Committed `.skill` zip contradicts `.gitignore` and has no freshness gate `[gap-fill]`
- **Files:** `.gitignore` (`*.skill` block); `three-loop-workflow.skill` (tracked, 63 KB);
  `three-loop-workflow/references/check-consistency.sh` (no zip handling).
- `.gitignore` forbids `*.skill` yet the zip is force-tracked. Git history shows a **reversed
  policy**: `207448b` untracked it ("built on demand and shipped via GitHub releases"), `d7c9b49`
  re-added it, and it's been hand-committed across 9 commits with the contradicting rule left in
  place. The zip is the actual distributed artifact (README: "package … and upload the .skill file"),
  but **no gate ties it to source** — `check-consistency.sh` never touches it; CLAUDE.md offers only a
  manual, unverified rebuild. A single forgotten rebuild ships a stale `SKILL.md` to consumers. It
  matches source today **by luck** (verified byte-for-byte), not enforcement. The `.gitignore`
  "GitHub releases" channel is fiction — no `.github/workflows`, no README mention.
- **Fix:** pick one — (A) `git rm --cached` the zip + build it in CI/release (matches the `207448b`
  rationale); or (B) keep it committed, delete the false `.gitignore` rule/comment, and add a
  `diff -rq` freshness gate wired into CLAUDE.md acceptance checks.

### H4 — No LICENSE on a public, install-me skill `[gap-fill]`
- **Files:** repo root (no `LICENSE`/`COPYING`/`NOTICE`); `README.md:9,53-81`; `SKILL.md:1-6`.
- Repo is PUBLIC, `licenseInfo=null`. README repeatedly invites copy / drop-in / package / upload;

- **Fix:** add a top-level `LICENSE` (MIT recommended — permissive, matches install-me intent, and

### M2 — `validate-commit-msg.sh` fails OPEN without jq `[gap-fill]`
- **File:** `validate-commit-msg.sh:23,34`.
- The jq-absent sed fallback copies the JSON command value **without unescaping**, so the standard
  `-m "…"` form (Claude Code's Bash tool always JSON-escapes the inner quotes as `\"`) extracts an
  empty `MSG` → line 35 passes every invalid commit. Reproduced in a jq-free PATH: jq path blocks,
  sed path allows, for `feat(phase2 round2): bad`, `feat(phaseX): bad scope`, `wip: invalid prefix`.
  The failing case is the common case; no degraded-mode signal.
- **Fix:** require jq with a clear warning (fail-closed or warn-and-exit), or JSON-unescape the
  fallback (`sed 's/\\"/"/g'`) and assert both paths agree; at minimum document that jq is required.

### M3 — `-am` / `--amend` / `-F` / `-C` extract empty `MSG` and pass unchecked (confirmed)
- **File:** `validate-commit-msg.sh:34-35`. `-am` has no literal `-m` substring → empty `MSG` → exit
  0. `-am` is an everyday form and not in the documented "not parsed" list.
- **Fix:** match `-[a-z]*m` in the extractor, or add `-am`/clustered short flags to the documented
  not-parsed list.

### M4 — None-tier load-bearing carve-out asks the author to self-certify "changes no rule" (confirmed)
- **Files:** `SKILL.md:24` and duplicated in `references/light-mode.md:21-24`.
- The None-on-load-bearing path turns on the author deciding pre-review that their edit is
  "non-commitment-clause / changes no rule" — exactly the author-asserted tier judgment the skill
  distrusts elsewhere (`SKILL.md:26` "fresh-eyes-enforced, not author-asserted"). For the
  highest-stakes file class the single None review isn't told to re-confirm.
- **Fix:** require the None reviewer to re-run the substantive/trivial test against the diff and route
  to Full on any commitment-clause touch. Patch **both** sites and extend line 26's clause to name the
  None reviewer.

### M5 — No standing gate on the `SKILL.md` word-count budget `[gap-fill]`
- **Files:** `check-consistency.sh` (no size assertion); baseline `SKILL.md` = 213 lines / 2883 words.
- v1.5 design explicitly chose `wc -w` as the anti-bloat metric and pinned 2883/2888 — but only as
  per-wave manual F-checks (AC-W1-G5 etc.), never promoted to a standing gate. No CI, no Makefile, no
  real hooks. The always-loaded surface can ratchet up undetected despite CLAUDE.md declaring
  anti-bloat "binding." AC-G6 is a banned-import + reference-file-count guard — it does **not** bound
  in-place file growth.
- **Fix:** add one assertion to `check-consistency.sh`:
  `WORDS=$(wc -w < "$SKILL/SKILL.md"); [ "$WORDS" -gt 2888 ] && { echo "BLOAT: $WORDS > 2888"; fail=1; }`
  Reuses the metric+value the design already settled — net-neutral codification.

### M6 — L3 phase runner has no resume / idempotency `[gap-fill]`
> **Correction (2026-06-16):** this finding (and the D3 fix originally shipped for it) **mis-stated the
> Workflow runtime**. Per the official docs (code.claude.com/docs/en/workflows) the runtime **journals
> every `agent()` result**, so a run **is resumable within the same session** — relaunch with
> `Workflow({scriptPath, resumeFromRunId})` and completed corners return cached results (no re-dispatch,
> no duplicate commits); the no-`Date.now`/no-`Math.random` rule exists precisely to make that replay
> deterministic. The restart-at-round-1 / duplicate-commit hazard applies **only across sessions** (Claude
> Code exited mid-run → next session starts fresh). The shipped `loop-3-workflow.md` "Resumption" section
> and the `l3-phase.js` comment were corrected to reflect this; "delete the dev branch before relaunch" is
> right **only** for a cross-session fresh start, not for in-session resume.
- **File:** `l3-phase.js` (state init + round-stable branch name); `loop-3-workflow.md` "Resumption".
- Phase state is process-local (`round`, `devResult`, `fixApplied`, …). **Within a session** the Workflow
  runtime journals every `agent()` result, so relaunching with `resumeFromRunId` replays completed corners
  from cache — no re-dispatch, no duplicate commits. **Across sessions** (Claude Code exited mid-run) the
  in-session journal does not carry over: the next session re-enters at `round = 1` and re-dispatches dev
  on the round-stable branch (`<phase>-dev-r1`), which would **stack duplicate commits** unless the stale
  branch is deleted first.
- Severity medium, **cross-session only** (blast radius = one phase's commits on a throwaway
  `--ff-only`-merged-then-deleted branch; re-run still yields a reviewed+accepted result).
- **Fix (shipped):** the `loop-3-workflow.md` "Resumption" section documents the within-session vs
  cross-session split — prefer `resumeFromRunId` within the session; before a cross-session relaunch the
  main agent deletes the prior dev branch (`git branch -D <phase>-dev-r1`) so the fresh dev does not stack
  duplicate commits. The `l3-phase.js` comment was scoped to match. (The original D3 fix and the body
  above prior to this correction over-stated non-resumability — see the Correction note at the top.)
